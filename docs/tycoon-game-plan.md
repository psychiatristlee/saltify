# 솔트빵 타이쿤 게임 기획서

> 작성일: 2026-05-08
> 목적: 솔트빵 IP를 활용한 두 번째 게임(시뮬레이션 타이쿤) 기획안.
> 배포 채널: 솔트빵 앱(현재 퍼즐게임 입점) 내 신규 모드 + 웹.

---

## 1. 게임 컨셉 (한 줄 정의)

**"홍대 1평 매대에서 시작해 글로벌 솔트빵 체인을 키우는 베이커리 운영 시뮬레이션."**

플레이어는 새내기 파티시에 시점에서 매장을 운영한다. 빵을 굽고, 손님을 맞고, 신메뉴를 R&D하고, 직원을 고용하고, 분점을 내고, 인플루언서·블로그·SNS로 마케팅한다. 실제 솔트빵의 메뉴와 매장(동교동 1호점)이 그대로 첫 챕터의 무대가 된다.

### 핵심 동사 (player verbs)
- **굽다 (Bake)** — 메뉴별 굽기 미니게임 / 큐 관리
- **팔다 (Sell)** — 손님 응대, 진열대 관리, 가격 책정
- **연구하다 (R&D)** — 신메뉴 개발, 시즌 한정, 토핑 조합
- **확장하다 (Expand)** — 매장 업그레이드, 분점, 본사화
- **알리다 (Promote)** — SNS 게시물, 블로그 포스팅, 협업

## 2. 코어 루프

```
하루 시작 (오전 11시)
   ↓
재료 수급 + 메뉴 결정 + 가격 책정
   ↓
영업 시뮬레이션 (1.5~3분 실시간 압축, 11:00~19:30)
  - 손님이 줄을 섬 → 진열대에서 픽
  - 진열대 비면 굽기 미니게임
  - 굽기 실패 → 손님 만족도 ↓
   ↓
하루 마감 (정산 화면)
  - 매출, 인기 메뉴, 남은 재고, 리뷰 코멘트
   ↓
저녁 페이즈 (R&D / 마케팅 / 채용)
   ↓
다음 날 시작
```

**한 사이클 = 약 5분** (모바일 세션 길이 고려). 7일 = 1주 챕터, 4주 = 1개월 결산.

### 메타 진행 (장기 동기)
- **레벨업** = 매장 등급 (홀-인-더-월 → 동네 명물 → 미슐랭 빕그루망 → 브랜드)
- **언락** = 신메뉴, 매장 인테리어, 직원 슬롯, 분점 슬롯, 마케팅 채널
- **랭킹** = 일/주/월 매출 글로벌 리더보드 (Firestore 기반)

## 3. 시스템 상세

### 3.1 메뉴 시스템
실제 솔트빵 메뉴를 그대로 시드 데이터로 사용 (`game/src/models/BreadType.ts` 와 공유). 신메뉴 R&D는 다음 트리:

```
Plain (튜토리얼) → Everything (토핑 트리) ─┬─→ Olive Cheese
                                        ├─→ Basil Tomato
                                        └─→ Garlic Butter (조합 트리) → Hotteok
                                                                    → ChocoBun (시그니처)
음료: Cold Brew → Cold Brew Latte → Milk Tea (페어링 음료)
```

각 메뉴는 `(굽기 시간, 마진, 손님 선호도, 시즌 보정)` 4개 파라미터로 모델링. 초코번처럼 마진은 낮지만 인기가 폭발적인 시그니처 메뉴는 광고탑 역할.

### 3.2 손님 시스템
4종 손님 페르소나 (실제 홍대 상권 기반):
- **출근족 (07:30~10:00)** — 빠른 응대, 음료 + 빵 1개
- **관광객 (11:00~15:00)** — 사진 찍기, 시그니처 선호, 객단가 ↑
- **카공족 (14:00~18:00)** — 음료 + 디저트, 체류 길어 좌석 점유
- **퇴근족 (17:30~19:30)** — 선물용 박스 구매, 객단가 최고

손님 만족도는 누적되어 **구글/네이버 평점** (별점 + 리뷰)으로 시각화. 평점이 떨어지면 손님 트래픽이 줄어든다.

### 3.3 굽기 미니게임
재료 → 반죽 → 발효 → 굽기 4단계. 각 단계마다 작은 인터랙션:
- **반죽** — 리듬 탭 (BPM에 맞춰 치대기)
- **발효** — 타이머 (미리 시작해두면 캐시처럼 동작)
- **굽기** — 온도 슬라이더 (메뉴별 적정 온도)
- **소금 토핑** — 드래그 (적정 양)

자동화 가능: 직원 고용 시 자동으로 미니게임 통과 (직원 스킬에 따라 품질 등급 ↓). 완전 idle 게임이 되지 않도록 핵심 미니게임은 항상 플레이어가 직접 할 때 보너스가 붙도록 함.

### 3.4 R&D / 신메뉴
- 일정 누적 매출 도달 → R&D 슬롯 언락
- 토핑/필링 카드 수집 (실제 솔트빵 토핑들: 깨, 양파, 올리브, 바질, 토마토, 초코, 매트차, 견과류, 시럽 등)
- 카드 2~3장 조합 → 신메뉴 시안 → 시식 평가 → 출시
- **이스터에그**: 잘못된 조합도 가끔 히트 (예: "참깨+초코" → 깨초코번)

### 3.5 직원
- 채용: 알바 / 정직원 / 파티시에 / 매니저 4티어
- 스킬: 굽기 속도, 응대 만족도, 충성도
- 충성도가 떨어지면 퇴사 → 일주일간 인력 부족
- 4명 정원이 차면 분점 매니저로 승진 가능

### 3.6 분점 / 확장
- 1호점 (동교동) → 2호점 (성수) → 3호점 (강남) → 글로벌 (도쿄/타이베이/뉴욕)
- 각 분점은 idle하게 매출을 발생시키지만, 매니저 스킬에 따라 효율 ↑↓
- 본점에서 신메뉴 출시 → 분점에 확산까지 시간차 (R&D 풀 공유 시스템)

### 3.7 마케팅 / 미디어
- **SNS 게시물 (인스타)** — 매일 1회, 자동/수동
- **블로그 포스팅** — 솔트빵 실제 블로그(blog 패키지)와 크로스링크: 게임 내 블로그 발행 → 실제 매장 트래픽 통계로 환산되는 척만 (메타-펀)
- **인플루언서 협업** — 가챠 형태로 등장, 비용 vs 트래픽 트레이드오프
- **네이버 광고 / 구글 광고** — 비용 대비 ROI 슬롯머신

### 3.8 이벤트 / 시즌
- 실제 캘린더와 동기 (5월: 어린이날 박스, 12월: 크리스마스 한정, 발렌타인: 초코번 한정)
- 라이브 이벤트는 Firestore에서 푸시되는 `events/active` 도큐먼트 참조 → 운영자(=실제 솔트빵 사장)가 원격으로 조정 가능

## 8. 기술 / 플랫폼 결정

### 8.1 요구사항
1. **iOS + Android + 웹** 3-플랫폼 동시 출시 (현재 솔트빵 앱 정책)
2. **실시간 시뮬레이션 + 미니게임** — 스프라이트 60fps
3. **수백 개 손님 동시 처리** — 비싼 DOM 렌더링은 부적합
4. **현재 퍼즐게임 코드베이스 재사용** — 인증/Firestore/쿠폰/i18n
5. **6개월 내 1차 출시 가능 규모**

### 8.2 옵션 비교

| 옵션 | 장점 | 단점 | 평가 |
|------|------|------|------|
| **A. 현 React + Vite + Capacitor 그대로** | 코드 100% 재사용. 인증/Firestore 즉시 사용. 학습 곡선 0 | DOM/CSS로는 60fps 손님 시뮬레이션 한계. 픽셀아트 렌더 불편 | △ 매장 1개까지는 OK, 분점부터 한계 |
| **B. React 셸 + 게임 화면만 PixiJS / Phaser** | 코드 재사용 + 게임 영역만 캔버스. WebGL 가속 | 두 시스템 브리지 코드 필요. 경계 관리 비용 | ◎ **추천** |
| C. Unity (WebGL + iOS/Android) | 업계 표준. 에셋스토어 풍부. 미니게임 제작 용이 | 새 프로젝트. 인증/Firestore 다시 붙여야 함. WebGL 빌드 무거움 (10MB+). 코드 공유 불가 | △ 풀 3D였다면 채택 |
| D. Godot 4 | 오픈소스, 가벼운 빌드. 2D 게임 강함 | iOS export 까다로움. Firebase 연동 직접 작성. 작은 커뮤니티 | × |
| E. Cocos Creator | 한국 시장 점유율 ↑, 모바일 최적 | 학습 자료 부족 (영문). 라이선스 모호 | × |

### 8.3 결정: **옵션 B — React + Vite + Capacitor 셸 + 게임 영역은 PixiJS**

**근거**:
1. 인증, Firestore CRUD, 쿠폰, i18n, 결제, 랭킹 — 이 모든 게 이미 `game/src/services/` 에 짜여 있음. Unity로 가면 0부터 다시.
2. PixiJS는 WebGL 기반이라 캔버스 한 장에 손님 50명, 빵 진열대 30개 그려도 60fps 무리 없음.
3. React-PixiJS 브리지: `@pixi/react` 또는 직접 `useEffect` 마운트. 사실상 게임 화면 한 컴포넌트만 PixiJS 캔버스로 띄우면 됨.
4. Capacitor가 PixiJS 캔버스를 그대로 네이티브 WebView로 감싸므로 추가 작업 0.
5. 미니게임(반죽/굽기)은 PixiJS 위에서 spine/lottie 애니메이션으로 충분.

**언제 옵션 C (Unity)로 전환하나**:
- 분점 50개 동시 idle 시뮬레이션이 WebView에서 끊길 때
- 3D 매장 인테리어 커스터마이징이 핵심 기능이 될 때
- 글로벌 동기 멀티플레이가 도입될 때

⇒ 이 셋 다 V2 이후 이슈. V1은 PixiJS로 충분.

### 8.4 디렉토리 구조 (제안)

```
game/
  src/
    puzzle/        # 기존 퍼즐 게임 (이전)
    tycoon/        # 신규
      pixi/        # PixiJS 씬, 스프라이트, 미니게임
      data/        # 메뉴, 손님, R&D 카드
      services/    # 매출 시뮬, 마케팅
      views/       # React UI (HUD, 정산창, R&D창)
    services/      # 공유: auth, firestore, coupon, i18n
    components/    # 공유: 로그인, 프로필, 랜딩
```

### 8.5 라이브러리 픽

| 용도 | 선택 | 비고 |
|------|------|------|
| 게임 렌더 | **PixiJS 8** | WebGL2/WebGPU, 가벼움 |
| React-Pixi 브리지 | `@pixi/react` v8 | hook 기반 |
| 애니메이션 | `gsap` (이미 익숙) + Spine 2D | |

## 10. 리스크 & 미해결 질문

1. **퍼즐게임 vs 타이쿤 — 한 앱 안에 둘 다? 별개 앱?**
   - 현재 의견: 한 앱 두 모드. 메인 화면에서 "퍼즐 / 타이쿤" 선택. 쿠폰 풀 공유.
   - 다른 의견: 타이쿤이 무겁다 (미디어 5MB+). 별개 앱으로 분리 가능.
   - **결정 필요.**

2. **실 매장 쿠폰 인플레이션** — 타이쿤은 매출 단위가 크므로 쿠폰 임계값을 어떻게 다시 설계하나? 퍼즐과 같은 비율이면 쿠폰 폭증 위험.
